1. What Bottlewatch does

Bottlewatch tracks binding bottlenecks in the AI supply chain — the chokepoints where the next leg of capacity is hardest to add — and the public companies whose share prices move first when those chokepoints tighten or break.

The thesis is straightforward: a static “what’s tight right now” scoreboard is consensus-priced by the time you see it. Edge lives in predicting trajectory — which bottlenecks are emerging (long, before consensus catches up), which are peaking (momentum long or trim), and which are resolving (short or skip; never long).

The system has three surfaces you actually use day-to-day:

  • A web scoreboard — Next.js dashboard, served at http://localhost:3000 in development.
  • A REST API — FastAPI service at http://localhost:8000/api/v1/.
  • Two launchd jobs that keep the data fresh (daily at 04:00 and 04:30).

Everything else — the ontology, the scoring seed, the segment briefs — is build-time material. You don’t need to read it to use the dashboard, but section 6 explains how the numbers get computed.

2. Install & first run

2.1 Prerequisites

  • macOS (the launchd jobs are macOS-specific; the rest of the toolchain runs anywhere Python 3.13 does).
  • uv — the Python package manager. Install with curl -LsSf https://astral.sh/uv/install.sh | sh.
  • Node 20+ / pnpm — for the dashboard. brew install node pnpm.
  • API keys for the data sources you want live (see 2.4).

2.2 Install the package

cd /Users/iljoyoo/workspace/bottlewatch
make sync          # uv sync — installs the package + dev group
make install-hooks # enables the .env pre-commit guard

The pre-commit hook refuses to stage any file named .env, so your API keys can’t leak even if you forget to update .gitignore entries.

2.3 Add your API keys

Copy the template and fill in the keys you have:

cp .env.example .env
$EDITOR .env

Keys in priority order:

VarWhat it enablesRequired?
EIA_API_KEYPower generation, capacity, electricity, and 860M planned-additions ingestsStrongly recommended
FRED_API_KEYTransformer PPI, power demand macro overlaysRecommended
COMTRADE_API_KEYHBM / transformer trade-flow ingestOptional

All three are free. Without a key, the corresponding adapter logs a warning and is skipped; the rest of the system runs fine on the remaining sources. URLs to register: EIA, FRED, Comtrade.

2.4 Build the ontology & run a one-shot refresh

make research    # ontology + validate-ontology — verifies the ABox builds
make ingest      # one-shot data refresh into data/processed/bottlewatch.db

The first ingest downloads several hundred MB of EIA capacity tables, so do it on a wired connection. Subsequent daily runs take a few minutes.

2.5 Start the API & the dashboard

Two terminals:

make api         # uvicorn at http://127.0.0.1:8000
make web         # pnpm dev at http://localhost:3000

Open the dashboard. You should see the scoreboard populated with the eleven segments. If the dashboard says “no data” for a segment, the ingest hasn’t completed yet for that source — see Troubleshooting.

2.6 Schedule the daily refresh

make schedule

This installs two launchd agents:

AgentRunsCommand
com.bottlewatch.refresh04:00 dailymake ingest
com.bottlewatch.recompute04:30 dailymake recompute

Both write structured JSONL output to data/cache/refresh.log. Refresh pulls the latest data; recompute rebuilds the score table from the updated signals.

To remove: make unschedule. To check status: launchctl list | grep bottlewatch.

The installer templates the project path into the plist so the same repo can be installed from any absolute location. If you ever move the directory, re-run make schedule to repoint the agents.

3. Daily operation

3.1 The day-to-day flow

You don’t need to touch anything on a normal day. At 04:00, the refresh job pulls new data. At 04:30, the recompute job rebuilds scores. By 05:00 the dashboard is showing fresh numbers.

When you sit down to look at it:

  1. Open the scoreboard. Skim the regime column for any segment that changed overnight — the dashboard highlights delta badges (arrow icons) when a regime flipped since you last looked.
  2. For any flagged segment, click through to the per-segment view. That view shows the five sub-scores, the 6-month momentum (B'), and the underlying signals that drove the change.
  3. Check the conviction basket. The screener endpoint returns the near/med/long baskets the model currently thinks are investable. The hard guard (no RESOLVING in the long basket) is non-overridable in the model — see section 6.4.
  4. Read or write thesis notes on any segment where you disagree with the regime call. The note is auditable: it shows up in the UI next to the guard, so future-you remembers why you overrode (or didn’t).

3.2 Backfilling

If you want scores further back than the last 6 months (the momentum window), use the backfill command:

make backfill

This calls bottlewatch-recompute --backfill-since 2024-01-01 and rebuilds the score history from the start of 2024. Run it once on a new install; rerun it only if you add a new signal source whose historical depth covers more than the live DB has.

3.3 What lives where

PathWhatGitignored?
data/raw/Immutable dumps from each sourceYes
data/processed/bottlewatch.dbSQLite — the runtime source of truthYes
data/cache/refresh.logJSONL refresh + recompute outputYes
research/Build-time inputs (CSV, briefs, ontology) — edit these to recalibrateNo
Never edit data/processed/bottlewatch.db directly. The next make ingest will overwrite it. Recalibrate the model by editing the research/ inputs and running make research + make recompute.

4. Reading the dashboard

4.1 The scoreboard

One row per segment. The columns are:

Segment
The supply-chain stage. Eleven of them — advanced-node fabs, advanced packaging, HBM, transformers, cooling, etc. See research/01_segments/ for the full list and what each covers.
Score (B)
A 0-100 tightness level for the selected horizon. Higher = tighter bottleneck. Computed from five sub-scores weighted by horizon; see section 6.1.
Momentum (B′)
The 6-month change in B, in the same units. Positive = tightening; negative = easing. Drives the regime call when combined with B.
Regime
One of six labels — see section 6.3. The short basket only considers RESOLVING; the long basket excludes RESOLVING outright.
ETA
Directional resolution estimate: <12mo, 12-24mo, or >24mo. Driven by the named capacity additions in research/06_capacity_ledger.md (manual ledger today; automated in v1.1).

4.2 The horizon toggle

Three buttons at the top of the scoreboard: near, med, long. Each has a different weight vector for the five sub-scores:

HorizonWindowWhat dominates
near~3-6 monthsLead times, current capacity tightness, today’s announcements
med~12-18 monthsCapacity additions, fab build-outs, demand growth
long~24-36 monthsGeographic concentration, regulatory friction, structural demand

The same B value can mean very different things across horizons. A segment scoring 75 in the near horizon is a crisis today; a 75 in the long horizon is a thesis that takes two years to play out.

4.3 The conviction basket

For each horizon, the screener view returns the top-3 to top-5 segments ranked by exposure_pct × B on the public tickers in research/02_universe.csv. The ordering is the model’s current best estimate of which segments have the highest risk-adjusted directional move per invested dollar.

The conviction basket is a starting point, not a buy list. It tells you which segments deserve the most attention. Within each segment, picking the specific ticker still requires reading the brief and the recent signals.

4.4 Thesis notes

On any segment, the pencil icon opens a markdown editor. Notes you write here are stored verbatim in the thesis table and appear in three places:

  1. Inline on the per-segment view, dated and timestamped.
  2. Attached to the conviction-basket row when the guard would fire (i.e., when the segment is RESOLVING but you’ve argued it’s not).
  3. Surfaced in the /api/v1/thesis endpoint for the screener UI.

Notes are the audit trail for any future model refinement. When the M4 backtest lands, the notes are the seed for figuring out which regime calls were correct in hindsight.

5. Reading the API directly

The dashboard is a thin layer over the API. When the dashboard doesn’t show what you need, hit the API directly. All endpoints are under http://localhost:8000/api/v1/ and return JSON.

MethodPathWhat you get
GET/healthDB connectivity, last-score timestamp, total signals. First thing to check.
GET/segmentsList of all segments with the latest score, regime, and ETA per horizon.
GET/segments/{slug}One segment, full detail: sub-scores, recent signals, ETA, thesis notes.
GET/segments/{slug}?horizon=nearSame, scoped to one horizon (also: med, long).
GET/signals?segment=...&limit=50The raw underlying signals driving a segment’s score. The proof layer.
GET/scores/regime?horizon=near2x2 quadrant payload, one row per segment, scoped to one horizon.
GET/scores/history?segment=hbm-memory&horizon=nearPer-segment B and B' time series for charting.
GET/tickersInvestable universe — one row per (ticker, segment) pair, annotated with current regime.
GET/tickers/{ticker}One ticker: segments it’s exposed to, regimes, exposures.
GET/screener?side=long&horizon=nearConviction basket for the long side. Hard guard: RESOLVING excluded. Also side=short, horizon=med|long.
GET/mapThe value-chain DAG as JSON, annotated with current regime per node. Feed to any graph viz.
GET/etaResolution ETA per segment with confidence (low / medium / high).
GET/POST/DELETE/thesis?segment=...Read, write, or delete thesis notes for a segment.

5.1 Health check

curl -s http://localhost:8000/api/v1/health | jq
# { "db_ok": true, "last_score_at": "2026-06-06T04:31:12Z", "signals_count": 147832 }

If db_ok is false, the API is up but the DB isn’t. Check data/processed/bottlewatch.db exists and that make ingest has been run at least once.

5.2 The screener (conviction basket)

curl -s 'http://localhost:8000/api/v1/screener?side=long&horizon=near' | jq

Returns an ordered list of segments. The hard guard is enforced server-side: even if your dashboard somehow misses the regime filtering, the screener will refuse to return a RESOLVING segment on the long side. Short side, by contrast, prefers RESOLVING segments ranked by B × |B'|.

5.3 Writing a thesis note (programmatically)

curl -s -X POST http://localhost:8000/api/v1/thesis?segment=hbm-memory \
  -H 'content-type: application/json' \
  -d '{"body_md": "HBM4 ramp is faster than the model assumes. SK hynix M15X line online May 2026."}'

The body is plain markdown. The frontend editor is a TipTap instance that round-trips through markdown on save.

6. Research & scoring model

This section explains the model. You don’t need to know it to use the dashboard, but you do need to know it to recalibrate, and you absolutely need to know it before you act on a regime call.

6.1 The formula

For each segment s and horizon h:

B(s, h) = 100 × sum_i  w_i(h) × normalize_5y(s_i)

Where s_i is a raw sub-score, normalize_5y maps it to [0, 1] over a 5-year rolling min/max band, and w_i(h) is the horizon weight for sub-score i (weights sum to 1.0 per horizon).

The five sub-scores:

lead_time_growth
How fast quoted lead times are rising year-over-year. Pulled from SEC 10-K capacity language for the named players.
capacity_tightness
Utilization vs. nameplate for the named capacity pools. The only sub-score computed from the live signals table; the rest are seeded from research/scoring_seed.json.
geo_concentration
HHI of the supplier-of-supplier pool for the segment. v1.1 will compute this from the ontology ABox via SPARQL; M2 uses a hand-coded value per segment.
regulatory_friction
Permitting / export-control / CHIPS-Act friction for adding capacity in the relevant jurisdictions. Hand-coded rubric.
demand_signal
Year-over-year growth in downstream demand, normalized. FRED PPI for transformers, EIA electricity growth for power, etc.

The formula’s three properties you can lean on:

  1. Bounded: B ∈ [0, 100] always. A 78 in HBM and a 78 in transformers mean the same thing.
  2. Horizon-aware: the same raw inputs can produce very different B values across horizons because the weight vector shifts.
  3. Auditable: every score decomposes into five numbers, and every signal has a source citation in research/. Drill from a score to a sub-score to a signal to a 10-K excerpt.

6.2 Momentum (B')

B' is the 6-month backward change in B, in the same units. It is the second axis of the regime classification. A high B tells you the bottleneck is tight; B' tells you whether it’s getting tighter or easing. Without momentum, you can’t tell PEAKING from PEAKED from RESOLVING — they all have high B.

6.3 The six regimes

Regimes are the actionable output. They are computed from (B, B') against the cell table in research/06_regime_thresholds.json:

RegimeWhat it meansAction bias
EMERGINGTightening from a low base — B is rising fastProactive long, before consensus catches up
PEAKINGHigh B, B' > 0 — still tightening, near the topMomentum long, or trim existing
PEAKEDHigh B, B' ≈ 0 — at the top, not yet easingHold; consider trim into strength
RESOLVINGHigh B, B' < 0 — easing from a high baseShort, or skip. Never long.
STABLEModerate B, low B'Skip — no edge
RESOLVING_FROM_LOWLow B, large negative B' — was never a bottleneck, is becoming less of oneSkip / short
The no_data_threshold in 06_regime_thresholds.json (default 0.4) is the data-completeness fraction below which a segment is marked NO_DATA rather than receiving any of the six regimes. NO_DATA segments are excluded from the screener — they are not investable on the model’s evidence.

6.4 The hard guard

The conviction basket enforces one non-overridable rule: no RESOLVING segment on the long side. The rule exists because the asymmetry is too clean to ignore. If a bottleneck is easing (B' < 0, high B), the segment’s pricing power is structurally declining — even if the absolute tightness looks attractive today. Going long RESOLVING is fighting the trajectory, not the level.

The guard is implemented in src/bottlewatch/app/api/screener.py. There is no override flag. If you disagree with a RESOLVING call, write a thesis note — it appears in the UI next to the guard line so future-you sees the disagreement, and the M4 backtest can grade whether the disagreement was right.

6.5 The value chain DAG

The dashboard’s “map” view shows the value chain as a directed acyclic graph, with each node colored by its current regime. The DAG is the source of one specific insight: bottlenecks travel along edges. A power segment with high B alone is in a different situation from one with high B and named capacity additions (GEV HA-class turbines, Three Mile Island restart, SMR commissioning) ramping 12-18 months out. The first is PEAKING; the second is PEAKED. The map’s job is to make the supplier-of-supplier edges legible enough that the regime call cites specific capacity additions, not vibes.

6.6 Where to recalibrate

The model’s parameters live in three places:

FileWhat it controlsWhen to edit
research/scoring_seed.jsonHand-coded sub-score values per segmentWhen a structural assumption changes (e.g., a new fab is announced)
research/06_regime_thresholds.jsonRegime cell boundariesRarely. M4 backtest may shift the B=70 boundary.
research/02_universe.csvTickers, segments, exposure_pctWhen a new public company joins a supply chain stage

After any change, run make recompute to rebuild the score table. The M4 backtest will measure whether the calibration actually improved regime-call accuracy against 12-18 months of hindsight.

7. Troubleshooting

“No data” for a segment in the dashboard

Check curl http://localhost:8000/api/v1/health. If signals_count is low, that source’s ingest hasn’t populated the table yet. Tail data/cache/refresh.log for errors. Common causes: missing API key, rate-limited source, network-bloked environment.

Refresh job is failing overnight

  1. Check the log: tail -f data/cache/refresh.log in JSON-pretty mode (jq helps).
  2. Check the agent status: launchctl list | grep bottlewatch.
  3. Run the job manually: make ingest and watch the output. The orchestrator logs a structured error per source on failure; the others continue.

Scoreboard regime flipped overnight and I disagree

  1. Open the per-segment view and read the recent signals. The B' arrow will show you which sub-score drove the flip.
  2. If the data is right and you still disagree, that’s a model call, not a bug. Write a thesis note. The M4 backtest will surface the disagreement.
  3. If the data is wrong (e.g., a stale 10-K figure, a misparsed EIA series), the right fix is in the adapter, not the threshold. File it as a bug in the project, and adjust scoring_seed.json as a temporary patch.

API is up but db_ok is false

SQLite file is missing or locked. The default path is data/processed/bottlewatch.db relative to the project root. ls -la it. If it’s missing, run make ingest. If it’s locked, find the process: lsof data/processed/bottlewatch.db.

Pre-commit hook blocks a commit

You tried to stage a .env file. Move the secrets to your shell environment or to .env outside the repo, and stage only the code change. The hook is non-overridable by design.

I need to add a new data source

That’s a developer task, not an operator task. The end-to-end procedure lives in the developer manual (TBD). The short version: write an adapter under src/bottlewatch/app/ingest/, register it in jobs/refresh_daily.py, add a column to app/score/extractors.py if the source feeds a new sub-score, and add tests under src/bottlewatch/tests/.
